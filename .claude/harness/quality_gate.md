# Quality Gate - 점수 기반 품질 게이트

> 긍정 편향을 수치로 차단하는 품질 검증 시스템

---

## 1. 점수 체계 개요

```
┌─────────────────────────────────────────────────────────┐
│  품질 게이트 = 영역별 점수 + 과락 기준 + 총점 커트라인  │
│                                                         │
│  ✗ Claude 주관적 판단 ("거의 완성", "잘 됐습니다")      │
│  ✓ 객관적 수치 (85/100점, 보안 19/20점)                 │
└─────────────────────────────────────────────────────────┘
```

**핵심 원칙**:
- Claude는 **실행자**, 판단은 **수치**가 한다
- 영역 하나라도 **과락**이면 총점 관계없이 **불통**
- 점수가 안 나오면 물리적으로 다음 단계 진행 불가

---

## 2. 영역별 배점 및 커트라인

### 2.1 표준 배점표 (100점 만점)

| 영역 | 배점 | 커트라인 | 과락 시 | 검증 방법 |
|------|------|----------|---------|-----------|
| **기능 동작** | 30점 | 24점 (80%) | 즉시 중단 | 실행 테스트, 요구사항 대조 |
| **보안** | 20점 | 18점 (90%) | 즉시 중단 | security-checker 감사 |
| **코드 품질** | 15점 | 10점 (67%) | 재작업 | lint, 타입검사, 컨벤션 |
| **UX/사용성** | 20점 | 14점 (70%) | 재작업 | ux-reviewer 감사 |
| **성능** | 10점 | 7점 (70%) | 경고 | 응답시간, 메모리 |
| **문서화** | 5점 | 3점 (60%) | 경고 | 주석, README, API문서 |
| **총점** | 100점 | **80점** | 불통 | 전 영역 합산 |

### 2.2 과락 정책

```
기능 동작 < 24점  →  🛑 즉시 중단 (배포 불가)
보안      < 18점  →  🛑 즉시 중단 (배포 불가)
코드 품질 < 10점  →  ⚠️ 재작업 필수
UX/사용성 < 14점  →  ⚠️ 재작업 필수
성능      < 7점   →  📋 경고 (선택적 개선)
문서화    < 3점   →  📋 경고 (선택적 개선)

※ 과락 영역이 1개라도 있으면 총점 관계없이 불통
```

## 3. 세부 채점 기준

### 3.1 기능 동작 (30점)

| 항목 | 배점 | 채점 기준 |
|------|------|-----------|
| 요구사항 완전 구현 | 12점 | 요청 기능 100% 구현 시 만점 |
| 기존 기능 미파손 | 8점 | 기존 코드 영향도 0 시 만점 |
| 에러 없이 실행 | 6점 | 런타임 에러 0개 시 만점 |
| 엣지케이스 처리 | 4점 | 빈값/null/경계값 처리 |

**채점 공식**:
```
기능_점수 = (구현된_기능수 / 요청_기능수) × 12
          + (영향받지_않은_기능수 / 기존_기능수) × 8
          + (에러_없음 ? 6 : 0)
          + (엣지케이스_처리율 × 4)
```

### 3.2 보안 (20점)

| 항목 | 배점 | 채점 기준 |
|------|------|-----------|
| 인젝션 방지 | 6점 | SQL/Command/XSS 인젝션 없음 |
| 인증/인가 | 5점 | 적절한 권한 검사 |
| 민감정보 노출 | 5점 | API키/비밀번호 하드코딩 없음 |
| 입력 검증 | 4점 | 사용자 입력 유효성 검사 |

**과락 즉시 불통 항목**:
- [ ] SQL 인젝션 가능성 → 0점 처리
- [ ] 하드코딩된 API키/비밀번호 → 0점 처리
- [ ] XSS 취약점 → 0점 처리

### 3.3 코드 품질 (15점)

| 항목 | 배점 | 채점 기준 |
|------|------|-----------|
| 문법 오류 없음 | 4점 | 파싱/컴파일 성공 |
| 타입 오류 없음 | 4점 | TypeScript/mypy 통과 |
| 린트 통과 | 3점 | ESLint/Pylint 경고 0개 |
| 코드 스타일 | 2점 | Prettier/Black 포맷 일치 |
| 중복 코드 없음 | 2점 | DRY 원칙 준수 |

### 3.4 UX/사용성 (20점)

| 항목 | 배점 | 채점 기준 |
|------|------|-----------|
| 빈 상태(Empty State) | 5점 | 데이터 없을 때 UI 제공 |
| 에러 상태 | 5점 | 에러 발생 시 친절한 메시지 |
| 로딩 상태 | 4점 | 비동기 작업 시 피드백 |
| 직관적 흐름 | 4점 | 사용자 혼란 없는 동선 |
| 접근성 | 2점 | 기본 접근성 준수 |

**UX 필수 체크리스트**:
- [ ] 빈 상태 화면이 있는가?
- [ ] 에러 메시지가 사용자 친화적인가?
- [ ] 로딩 인디케이터가 있는가?
- [ ] 버튼/링크가 명확한가?

### 3.5 성능 (10점)

| 항목 | 배점 | 채점 기준 |
|------|------|-----------|
| 응답 시간 | 4점 | 목표 시간 내 응답 |
| 리소스 사용 | 3점 | 메모리/CPU 적정 |
| 최적화 | 3점 | 불필요한 연산 없음 |

### 3.6 문서화 (5점)

| 항목 | 배점 | 채점 기준 |
|------|------|-----------|
| 코드 주석 | 2점 | 복잡한 로직 설명 |
| API 문서 | 2점 | 엔드포인트/함수 문서화 |
| README | 1점 | 사용법 명시 |

## 4. 단계별 기준 상향

프로젝트 성숙도에 따라 기준을 점진적으로 상향:

| Phase | 총점 커트라인 | 특징 |
|-------|--------------|------|
| **Phase 1 (초기)** | 70점 | MVP 단계, 기능 동작 위주 |
| **Phase 2 (안정)** | 80점 | 영역별 커트라인 엄격 적용 |
| **Phase 3 (고도화)** | 85점 | 항목 세분화, UX 강화 |
| **Phase 4 (완숙)** | 90점 | 전 영역 고품질 요구 |

### Phase 전환 기준

```
Phase 1 → 2: 연속 5개 프로젝트 70점 이상 달성
Phase 2 → 3: 연속 5개 프로젝트 80점 이상 달성
Phase 3 → 4: 연속 5개 프로젝트 85점 이상 달성
```

## 5. 점수 산출 프로세스

### 5.1 평가 흐름

```
작업 완료
    ↓
evaluate_agent 호출
    ↓
┌─────────────────────────────────┐
│ 영역별 점수 산출                │
│ 1. 기능 동작: XX/30             │
│ 2. 보안: XX/20                  │
│ 3. 코드 품질: XX/15             │
│ 4. UX/사용성: XX/20             │
│ 5. 성능: XX/10                  │
│ 6. 문서화: XX/5                 │
└─────────────────────────────────┘
    ↓
과락 체크
    ↓
┌─ 과락 있음 → 🛑 즉시 불통 + 해당 영역 재작업
│
└─ 과락 없음
        ↓
    총점 계산
        ↓
    ┌─ 총점 < 커트라인 → ⚠️ 불통 + 전체 검토
    │
    └─ 총점 ≥ 커트라인 → ✅ 통과 + 배포 승인
```

## 6. 불통 시 재작업 프로토콜

### 6.1 재작업 메시지 형식

```
[품질 게이트 불통 - 재작업 요청]

총점: 72/100 (커트라인: 80)
결과: FAIL

영역별 점수:
✅ 기능 동작: 26/30 (커트라인 24 통과)
✅ 보안: 18/20 (커트라인 18 통과)
❌ 코드 품질: 8/15 (커트라인 10 미달) ← 과락
⚠️ UX/사용성: 12/20 (커트라인 14 미달)
✅ 성능: 8/10 (커트라인 7 통과)
✅ 문서화: 4/5 (커트라인 3 통과)

과락 영역 상세:
1. 코드 품질 (8/15):
   - 타입 오류 3개 발견: src/api.ts:45, src/utils.ts:12, src/handler.ts:78
   - 린트 경고 5개: unused variables
   → 수정 필요: 타입 오류 해결, 미사용 변수 제거

2. UX/사용성 (12/20):
   - 빈 상태 화면 없음 (0/5)
   - 에러 메시지 불친절 (2/5)
   → 수정 필요: Empty State 컴포넌트 추가, 에러 메시지 개선

재작업 지시:
위 과락 영역을 수정하여 다시 제출해주세요.
```

### 6.2 재시도 정책

| 시도 | 조치 |
|------|------|
| 1차 불통 | 과락 영역 명시 + 구체적 수정 지시 → 재작업 |
| 2차 불통 | 다른 접근 방식 제안 + 관련 이슈 참조 → 재작업 |
| 3차 불통 | 사용자에게 보고 + 수동 개입 요청 |
